## 一、项目概述

费曼伴学智能体（考研专属）是基于费曼学习法的 AI 伴学工具：AI 扮演"零基础小白听众"，通过连续追问倒逼学习者主动输出讲解，定位知识盲区，最终输出量化诊断报告。

当前处于 **MVP 验证阶段**：第三周已完成单知识点（Dijkstra 算法）的费曼对话闭环；第四周目标为**教材知识点动态解析**与 **LangGraph 追问流程重构**，使系统从"硬编码单知识点"演进到"上传教材即可自动拆解知识点并生成四维 rubric"。

## 四、关键功能实现逻辑

### 4.1 费曼对话状态机（当前实现）

入口 `FeynmanService.chat()`，按顺序判定：

1. **偏题检测**（`_is_off_topic`）：命中"天气/吃饭/电影…"且不含 Dijkstra 主题词 → `guide_topic`，不计追问轮次。
2. **无效回答检测**（`_is_ineffective_answer`）：输入"不会/不知道"或长度≤2 → 给线索式追问；连续 2 次无效 → 给关键词式提示。不计追问轮次。
3. **追问轮次判定**：`follow_up_count < 3` → 走 LLM 评估；≥ 3 → 走 Mock 兜底直接生成报告。
4. **LLM 评估 + 兜底**（`_evaluate_with_fallback`）：先调 DeepSeek；失败自动降级到 MockLLM，记录 `last_provider` 和 `fallback_used`。
5. **契约归一化**（`_normalize_contract`）：`follow_up`/`guide_topic` 时强制 `card_preview`、`final_report` 为 null；`generate_report` 时强制两者非空。

### 4.2 DeepSeek 调用与结构化输出

- **System Prompt**（`prompts.py`）：注入"零基础小白"人设 + Dijkstra 四维 ground truth + 轮次规则 + 行为规则 + JSON Schema 约束。
- **User Prompt**（`build_user_prompt`）：拼接历史对话（最近 8 条）+ 当前轮次 + 用户输入。
- **结构化输出**：`response_format={"type": "json_object"}` 强制 JSON；`_parse_json_object` 做正则兜底提取。
- **温度 0.2**：降低随机性，保证评判稳定。

### 4.3 Mock 兜底客户端

DeepSeek 不可用或调用失败时启用，保证 demo 不中断：

- **关键词覆盖度**（`_coverage`）：检测用户讲解是否覆盖"非负权/贪心/松弛/证明"四个核心点。
- **追问逻辑**：按四个点未覆盖顺序发起针对性追问。
- **打分逻辑**：按覆盖点数 + 文本长度，计算四维分数（每维 0-10，总分 0-40）。
- **报告生成**：总分≥32 理解较完整、≥22 掌握流程需补强、否则关键概念需重建。

### 4.4 四维诊断报告

`generate_report` 时返回：
- `card_preview`：总分（0-40）+ 一句话摘要。
- `final_report`：四维（理解深度/表达完整性/逻辑连贯性/结构化能力）各自的得分、漏洞分析、改进建议 + 整体评价。
- 前端用 ECharts 雷达图展示四维，配合分维度文字。

### 4.5 会话管理

- 前端页面初始化生成 `session_id`（uuid），全流程复用。
- 后端内存存储，记录 `follow_up_count`、`invalid_answer_count`、`off_topic_count`、`ended`、`last_provider`、`fallback_used`。
- `reset` 接口清空状态；调试接口 `/session/{id}` 可查会话内部计数。

## 五、第四周技术路径说明：教材动态解析为什么不用 RAG

### 5.1 核心判断

第四周的"上传 PDF → 自动拆知识点 → 生成四维 rubric"功能，**V1 主路径不使用 RAG（切片+向量化+语义检索）**，原因如下：

1. **用户流程含"选章节→选知识点"交互**：人选定了知识点，等于已经替 LLM 选定了用哪块原文、哪个 rubric，**不需要语义检索来定位信息来源**。 只有当我们支持 用户随便问一句话/用“对话”的方式指定agent考他哪个知识点，需要RAG（**语义检索**）
2. **定位靠章节结构，不靠语义**：PDF 自带目录可解析出章节归属，chunk 按章分桶是确定性的，不需要"从大池子里语义搜"。
3. **单章/单知识点的原文塞得进上下文**：DeepSeek 上下文 64K-128K token，一章几千字直接进 prompt，grounding 就完成了。

### 5.2 防幻觉手段（不依赖 RAG）

| 手段 | 说明 |
|---|---|
| grounding | 把该知识点的原文 chunks 塞进 prompt |
| 结构化输出 | Function Calling/json_object 强制 JSON |
| 引用约束 | rubric 每条结论标注来源 chunk_id + 页码 |
| 验证 pass（V1.5） | 二次 LLM 检查结论是否被 chunk 支持 |

### 5.3 RAG 真正触发的条件（V1 不触发）

| 触发条件 | 场景 |
|---|---|
| 目录坏了无法按章分桶 | 需要语义检索定位 chunk 归属 |
| 追问时动态引用原文原句溯源 | 需要 runtime 语义检索 |
| 跨几百知识点语义推荐 | 大规模语义检索 |

以上三条 V1 均不触发（限定"有目录的文字版 PDF"），故不上向量库。

### 5.4 关键概念区分

| 概念 | 是什么 | V1 是否用 |
|---|---|---|
| 解析 | PDF→结构化文本（纯代码） | 用 |
| 切片 | 长文档切小块（纯代码），按页分块 | 用 |
| grounding | 原文塞进 prompt 防幻觉 | 用 |
| RAG | 切片+向量化+语义检索 | 不用 |
| 向量库 | 语义检索引擎 | 不用 |
